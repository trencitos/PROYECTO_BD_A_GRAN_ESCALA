1. TECH STACK
--------------------------------------------------------------------------------
* Workflow Orchestration: Apache Airflow (Python-based DAG)
* Data Processing & ETL: Apache Beam (Python SDK) with PyArrow for Parquet output
* Data Modeling & Transformation: dbt (Data Build Tool)
* Analytics Layer Database: PostgreSQL 15 (runs inside Docker container)
* Containerization: Docker Compose
* Data Storage Formats:
  - Bronze (Raw): CSV format files (data/sales_data.csv, data/status_logs.csv)
  - Silver (Enriched): Parquet files with nested structs and list schemas
  - Gold (Analytic): PostgreSQL database tables / views (supermarket_db)

3. CODING CONVENTIONS & DESIGN PATTERNS
--------------------------------------------------------------------------------
* Pipeline Flow (ETL/ELT):
  - Ingestion & cleaning happens in Python/Beam (ETL to Silver Parquet).
  - Transformation & aggregation happens in dbt/SQL (ELT to Gold DB).
* Airflow DAGs:
  - Simple bash command-driven DAG using `BashOperator` to execute Beam/dbt tasks.
  - Resilience configurations set in defaults: 2 retries, 5 min delay.
  - Employs a `FileSensor` to detect if files exist in the Silver layer directory.
* Apache Beam Pipeline:
  - Structured Object-Oriented pipeline utilising custom Apache Beam `DoFn` classes.
  - Tagged Side Outputs (`TaggedOutput`) to gracefully isolate invalid data.
  - Schema Enforcement: Leverages PyArrow Schema objects (`SILVER_SCHEMA`) to structure
    denormalized datasets with nested maps and arrays before exporting to Parquet.
* dbt/SQL Transformations:
  - Employs Common Table Expressions (CTEs) for readability and modular design.
  - Custom data parsing is utilized (specifically PG `UNNEST` over JSON/nested structs).
  - Standardized metrics (e.g. converting sales local currencies PEN, CLP, ARS to USD).
  - Integrates data quality assertions (`unique`, `not_null`, and custom range checking
    using the `dbt_expectations` package).

4. EXISTING MODULES AND FEATURES
--------------------------------------------------------------------------------
* Data Quality & Ingestion Layer:
  - Source file loading from `data/*.csv`.
  - Sales validators ensure row structure length, not-null key identifiers (ID, SKU),
    numeric and strictly positive transaction amounts.
  - Log validators restrict status entries to CREATED, PENDING, COMPLETED, REFUNDED.
  - Isolation of faulty records to `rejected_sales.csv` for audit trail.
* Integration & Denormalization Layer:
  - Integrates sales records with status history streams by merging inputs on
    Transaction ID (`CoGroupByKey`).
  - Forms a nested schema mapping representing individual transactional actions along
    with their detailed status transition arrays.
  - Outputs to standard compressed Parquet files (`silver_layer/sales_enriched`).
* Aggregation & Analytic Modeling Layer:
  - Stage layer: Extracts elements directly from the Parquet structs.
  - Seed mapping: References currency exchange mappings stored in `ref_exchange_rates.csv`.
  - Facts mapping: Derives creation-to-completion duration (`conversion_time_hours`),
    checks if a transaction was ever refunded (`is_refunded`), and normalizes amounts to USD.
  - Integration tests validating transaction uniqueness and validating that USD amounts
    fall inside valid positive boundaries.
